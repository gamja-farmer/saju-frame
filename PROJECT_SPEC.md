# PROJECT_SPEC.md  
대만 대상 사주팔자 사이드 프로젝트 스펙 정리

---

## 1. 프로젝트 개요

- 프로젝트 주제: 사주팔자(四柱八字) 기반 운세/해석 서비스
- 목표: 월 약 10만 원 수준의 소규모이지만 실제 수익 발생
- 성격: 개인 사이드 프로젝트, 저비용·저위험 실험
- 핵심 철학:  
  - “사주 결과”보다 “사주를 이해하게 만드는 해석 프레임”  
  - 정확함보다 납득감·공감·체류 시간 중시

---

## 2. 타깃 시장

- 국가: 대만
- 언어: 번체 중국어 (Traditional Chinese)
- 문화적 전제:
  - 명리/사주에 대한 거부감이 낮음
  - 텍스트 기반 해석에 익숙
  - 감정·관계·인생 흐름 중심 콘텐츠 선호
- 주 사용자:
  - 20~40대
  - 연애, 커리어, 인간관계 고민이 많은 층
  - “정확한 계산”보다 “나를 이해해주는 설명”에 반응

---

## 3. 벤치마크 & 차별화

### 3.1 벤치마크

- 한국의 청월아씨 사주 페이지
- 참고 요소:
  - 질문 → 공감 → 해석으로 이어지는 흐름
  - 결과를 단정하지 않고 해석의 여지를 남기는 문장
  - 감성적이지만 과도하지 않은 UX

### 3.2 대만 기존 사주 사이트의 한계

- 입력 즉시 결과 출력 (몰입도 낮음)
- 한 화면에 정보 과다
- 표·한자·전문 용어 중심
- “맞다/틀리다” 판단을 유저에게 강요

### 3.3 본 서비스의 차별점

- 즉답형 결과 배제
- 해석을 단계적으로 분해
- “당신이 이런 사람일 가능성이 높다”는 구조
- 스스로 고개를 끄덕이게 만드는 UX

---

## 4. 핵심 UX 전략

1. 계산 결과를 앞에 두지 않는다  
2. 사용자의 상태를 먼저 언어화한다  
3. 사주 용어는 뒤로 숨기고 해석을 전면에 배치한다  
4. 모든 문장은 “아마도 / 경향적으로 / 이런 흐름을 보인다” 형태 유지  
5. 체류 시간이 곧 신뢰도라는 전제

---

## 5. 페이지 구조 (텍스트 와이어프레임 기준)

### 5.1 진입 페이지 (Landing)

- 헤드라인  
  - “你是不是也有這種感覺？”
- 공감 문장 2~3줄
- 이 서비스가 해주는 것
  - 성격을 단정하지 않음
  - 인생의 흐름을 설명함
- CTA
  - “查看你的命盤解讀”

---

### 5.2 정보 입력 페이지

- 최소 입력:
  - 생년월일
  - 출생 시간 (선택)
  - 성별 (선택)
- 입력 중간중간 설명 문구 삽입
  - “不知道也沒關係”
- 버튼:
  - “開始解讀”

---

### 5.3 해석 결과 페이지 (핵심)

1단계: 전체 인상  
- “你的命盤給人的第一個感覺是…”

2단계: 성향 프레임  
- 에너지 사용 방식
- 관계에서의 패턴
- 선택을 미루는/빠른 이유

3단계: 삶의 흐름  
- 강해지는 시기
- 흔들리는 시기
- 반복되는 테마

4단계: 사주 용어 보조 설명 (접기/펼치기)
- 오행
- 십성
- 간단한 관계 설명

---

## 6. 사주 로직 설계 원칙

- AI 사용 최소화
- 외부 API 의존 배제
- 규칙 기반 / 계산 기반 구조
- 사주 타입은 약 10~15개 내외로 제한
- 결과 다양성은 “문장 조합”으로 확보

---

## 7. 콘텐츠 전략 (SEO)

- 공감형 블로그 글 → 결과 페이지 유입
- 예시 주제:
  - “為什麼你總是做決定很慢？”
  - “命盤裡容易累的人有什麼特徵？”
- 글 구조:
  - 상황 공감
  - 일반적 설명
  - 사주 관점 해석
  - 서비스 CTA 연결

---

## 8. 수익화 전략

### 8.1 초기

- 페이지 하단 광고 배너
- 과도한 노출 금지
- 신뢰도 우선

### 8.2 추후 가능성

- 상세 해석 잠금 해제 (소액)
- 특정 테마 유료 해석
- 광고 제거 옵션

---

## 9. 성공 판단 기준

- 방문 대비 결과 페이지 도달률
- 평균 체류 시간
- 재방문율
- “정확하다”보다 “이해된다” 반응

---

## 10. 핵심 요약 문장

- 이 서비스는 사주를 맞히는 서비스가 아니다  
- 사용자가 자기 인생을 설명할 언어를 빌려주는 서비스다  

---

## 11. 현재 구현 계획

논의를 통해 확정된 설계와 우선순위. 구현 시 이 섹션을 기준으로 진행한다.

### 11.1 현재 구현 상태 요약

| 영역 | 상태 | 비고 |
|------|------|------|
| 라우팅·i18n | 완료 | `[locale]`, input, result/[type], blog, zh-TW/ko/en 메시지 |
| 사주 계산 | 골격만 | calculator.ts 규칙 있음, calendar.ts 양력=음력 반환 (TODO) |
| 분류 | 완료 | classifier.ts → 10개 SajuType |
| 해석 조합 | 완료 | composer.ts + locale별 템플릿 |
| 입력 페이지 | UI만 | input 폼 마크업·번역만, submit/연동 없음 |
| 결과 페이지 | 더미 데이터 | result/[type] 고정 pillar 사용, 실제 입력과 미연결 |
| 해석 문장 | 비어 있음 | zh-TW 템플릿 빈 배열 → 결과 "—" |
| 랜딩 | 최소 | 헤드라인 + CTA만 |
| 블로그 | 구조만 | 목록/슬러그 있음, 글 없음 |

### 11.2 확정 설계

- **결과 공유**: 생년월일 제외. (type, variantIndex) 만 URL. 타입당 K개 변형 미리 정의, 정적/세그먼트 기반. 공유 시 동일 URL = 동일 결과.
- **K**: 5. 타입당 5개 변형(0~4), 전체 50개 결과 세트.
- **결과 URL**: `/result/[type]/[v]` (세그먼트). v 없이 접근 시 → 입력 유도 리다이렉트.
- **콘텐츠 구조 (방향 2)**: 기존 4배열(impression, tendency, flow, terminology) 유지. (type, v)로 세트를 조합해 반환하는 레이어(getVariantContent 등)만 추가.
- **음력 변환**: **lunar-typescript** npm 라이브러리. `src/lib/calendar.ts`의 solarToLunar를 해당 라이브러리 호출로 대체 예정.

### 11.3 작업 항목 및 우선순위

1. **C. 해석 콘텐츠 채우기 (zh-TW)** — 1순위. 10타입×K(5) 변형에 맞게 4단계 문장 추가, 방향 2 인덱스 설계.
2. **A. 입력→결과 연결**: 폼 연동, Server Action → `/result/[type]/[v]` redirect, getVariantContent 조회, v 없을 때 리다이렉트.
3. **B. 음력 변환**: lunar-typescript 도입, calendar.ts 교체.
4. **D. 랜딩·결과 UX**: 공감 문장·서비스 설명, 결과 페이지 용어 접기/펼치기.
5. **E. 블로그·SEO**: zh-TW 공감형 1~2편, CTA 연결.

### 11.4 확정 사항 요약

| 항목 | 확정 내용 |
|------|-----------|
| 결과 공유 | (type, variantIndex) 만 URL, K개 변형, 정적/세그먼트 |
| K | 5 |
| 1순위 | C (해석 콘텐츠) |
| 결과 URL | `/result/[type]/[v]` |
| v 없이 접근 | 입력 유도 리다이렉트 |
